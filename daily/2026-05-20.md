## 上午：Context + RAG + Agent

> Handbook: Context / RAG / Agent

### Context（上下文）

**第一性原理**：模型只能基于它看见的上下文行动；系统必须决定什么能进上下文、带着什么身份、过期后怎么退出。

- **Context Window**：容量 ≠ 理解力
- **Context Engineering**：信息分层——系统指令 > 链上事实 > 用户意图 > 外部网页
- **Memory**：过去允许 ≠ 现在允许，涉及资产和权限必须重新授权
- **Knowledge Base**：需要来源、版本、更新时间、废弃状态

金句：「Context 是信息治理，不是长文本拼接。」

### RAG（检索增强生成）

**第一性原理**：RAG 的可靠性取决于证据链，不取决于向量库。

证据链八环节：文档来源 → Chunking → Vector DB → Retriever → Rerank → 进入 Context → Generation → Citation

- **Chunking**：按结构切，保留版本 metadata
- **Retriever**：不只是向量相似度，要 filter(version="V4")
- **Citation**：答案必须追溯回具体文档/段落

金句：「没有 citation 的 RAG，只是把幻觉从模型内部搬到了检索系统里。」

### Agent（智能体）

**第一性原理**：Agent 不是自主权，是被约束的执行循环。

- **Tool Use**：读数据 vs 写数据库 vs 支付接口，风险完全不同
- **Planning**：候选路线，不是授权
- **State**：外置、可查询、可审计
- **Reflection**：自我检查提高质量，但不能替代外部验证
- **Multi-Agent**：放大协调问题

金句：「最危险的设计 = 模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限。」

### Pipeline

```
Knowledge Base → RAG(检索+引用) → Context(分层放置) → Agent(读状态+调工具+验证)
```

### Quiz — 5 题串联（跳过答题，直接讲解）

1. Context 信任分层：dApp 官网声明 / 用户推荐 / 链上信号 不能平铺在同一层
2. RAG 搜到但不对：Chunking 缺版本 metadata + Retriever 只做纯向量搜索
3. Agent 自我检查边界：自我反思 ≠ 确定性验证
4. Memory 隐性风险：Memory 不能替代实时授权
5. 三章串联：Context 分层 / RAG citation / Agent 只读 vs 写入 各自的关键设计问题

## 下午：MCP + Frameworks + Vibe Coding

### MCP（模型上下文协议）

**第一性原理**：模型不应直接拥有世界；应通过明确协议访问被授权的上下文和工具。

- **Server**：暴露 resources/tools/prompts，设计重点是边界
- **Client**：发现 server 能力，把工具信息交给模型
- **Tool Schema**：不只是字段类型——要说明什么时候用、参数含义、是否修改外部状态、是否需要确认
- **Permission**：MCP 使工具连接方便 ≠ 安全。区分只读/写入、会话/长期授权

Web3 分工：「MCP 负责工具发现和调用格式；Web3 账户系统负责最终权限和执行边界。」

### Frameworks（框架）

**第一性原理**：框架是系统边界的表达，不是智能本身。先理解工作流，再决定用不用框架。

| 框架 | 管什么 |
|------|--------|
| LangChain | 组件库：模型接入、prompt、工具、retriever、agent |
| LangGraph | 状态机：多步任务、重试、人工确认、分支、恢复 |
| OpenAI Agents SDK | Agent 工程套件：handoff、guardrails、tracing |
| DSPy | 程序化优化 prompt/pipeline：有数据+指标 → optimizer 自动改进 |
| Learning Agent | 从反馈改进：记录失败 → 标注 → 入 eval → 改配置 → 测试上线 |

核心判断：「简单链路先保持简单。框架要能退出。」

### Vibe Coding（氛围编程）

**第一性原理**：AI Coding 的核心不是自动生成代码，而是把工程反馈循环压短。

公式：「任务要小 + 上下文要准 + 验证要硬 = 有效 Vibe Coding」

五层阶梯：Vibe Coding → Claude Code/Codex CLI → Model/API Config → GitHub/gh + CLI/Script → Repo Workflow

Web3 红线：「链上相关代码不能只靠 Agent 生成后直接上线。」

### Quiz — 4 题（MCP schema / 不该用框架 / 任务边界 / 三章串联）

1. MCP tool schema：参数类型 + 副作用标注 + requires_confirmation > 一个 string 字段
2. 8h Hackathon 不需要 LangGraph state machine → 简单链路保持简单
3. Agent 改了不该改的 slippage 代码 → 任务不够小 + 没有硬验证
4. 合约风险分析 Agent 各层分工：MCP=工具发现、Frameworks=状态管理、Vibe Coding=开发迭代

## 晚上：Evaluation + Fine-tuning + Inference

### Evaluation（评估）

**第一性原理**：不能被重复测量的 AI 行为，就不能被稳定改进。

| 概念 | 做什么 |
|------|--------|
| Harness | 喂样本、调系统、收输出、跑 grader、记结果。可重复是核心 |
| Golden Set | 30-100 条精心标注样本，覆盖正常/边界/易错/高风险/历史 bug |
| LLM-as-Judge | 用模型评模型，但不能神化。规则 > LLM judge > 人工抽检 |
| Regression | 防止修 A 坏 B，每次改动重跑历史问题集 |
| Observability | 线上观察——用户在哪里失败、该往 golden set 补什么 |

### Fine-tuning（微调）

**第一性原理**：改的是模型行为分布，不是产品系统边界。

- **SFT**：监督微调，让模型学格式/风格/任务流程
- **LoRA**：参数高效微调，训小 adapter，降成本
- **Dataset**：训练集/验证集/测试集/回归集。不要把测试集拿去训练
- **Overfitting**：记训练数据太死，真实用户一问就崩

核心流程：「先有 eval → 再修数据 → prompt/few-shot 搞不定 → 才微调。」

### Inference（推理服务）

**第一性原理**：在约束条件下交付可用答案。不是选模型，是选一套延迟/成本/质量/隐私/运维的平衡。

| 层 | 做什么 |
|----|--------|
| API Model | 调云端 API，但要处理限流/超时/重试/脱敏/账单 |
| Local Model | 本地跑——隐私、离线、低成本。吃硬件 |
| Quantization | FP16→INT8/INT4，降显存。可能降质量，要实测 |
| Serving | 并发、队列、流式、缓存、监控、灰度、降级 |

Web3 要求：「链上动作不可逆，推理层必须留可审计记录。」

## 收获

### 核心心智模型

```
Prompt → Structured Output → Context(分层) → RAG(证据链) → Agent(约束执行)
  → MCP(工具标准化) → Frameworks(工程组织) → Vibe Coding(开发迭代)
    → Eval(可测量改进) → Fine-tuning(行为分布) → Inference(交付约束)
```

### 关键工程直觉

1. **信任分层**：每段信息必须标注来源身份和信任级别，不能混在同一层
2. **证据链**：RAG 的可靠性不取决于向量库，取决于 chunk 版本 + metadata filter + citation
3. **约束执行**：Agent 的安全红线——自我反思 ≠ 安全校验，Memory ≠ 授权，Planning ≠ 执行许可
4. **协议化**：MCP 把工具连接标准化，但权限和审计仍由系统实现
5. **框架边界**：先理解工作流再选框架，简单链路保持简单，框架要能退出
6. **任务要小**：Vibe Coding 的有效性取决于任务边界 + 上下文准确性 + 硬验证
7. **先有 eval**：不能被测量的行为就不能被稳定改进。fine-tune 前必须先有 eval
8. **推理可审计**：链上动作不可逆，推理层必须留记录——哪个模型、什么输入、什么输出

### 三个安全红线（贯穿全章）

- 自我反思不能替代外部校验
- Memory 不能替代实时授权
- 框架/AI Coding 不能替用户承担资产风险

### Quiz — 4 题（Eval 优先级 / Fine-tuning 先后顺序 / Inference 选择 / 三章闭环）

1. Golden set 优先加高风险误判样本（B），而非格式缺陷（A）。Handbook：「先保住关键失败场景。」
2. Fine-tune 前：先修 prompt/context/检索 → 建立 eval golden set → 证明 prompt 修不好再微调。
3. Local Model：数据不能离开服务器 → API Model 出局。1000次/天 + <3s，量化本地模型够用。
4. 闭环：Eval 加入 regression set → Fine-tuning 修行为分布 → Inference 记录模型版本和输入输出。
