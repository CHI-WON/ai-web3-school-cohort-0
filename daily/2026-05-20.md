## 学习内容

### 1. 上下文（Context）

> Handbook: https://aiweb3.school/zh/handbook/ai-basics/context/

**第一性原理**：模型只能基于它看见的上下文行动；系统必须决定什么能进上下文、带着什么身份、过期后怎么退出。

核心概念：
- **Context Window**：容量 ≠ 理解力。塞得多不等于模型会用。
- **Context Engineering**：信息来源分层——系统指令 / 任务目标 / 链上事实 / 检索文档 / 用户记忆，各自明确身份和信任级别。
- **Memory**：跨请求记住偏好很方便，但过去允许 ≠ 现在允许，涉及资产和权限必须重新授权。
- **Knowledge Base**：外部知识库需要来源、版本、更新时间、废弃状态。

关键金句：Context 是信息治理，不是长文本拼接。把"用户说的""网页写的""链上查到的""系统规定的"混在同一层处理 = 灾难。

### 2. 检索增强生成（RAG）

> Handbook: https://aiweb3.school/zh/handbook/ai-basics/rag/

**第一性原理**：RAG 的可靠性取决于证据链，不取决于向量库这个名词。

核心概念：
- **Chunking**：按结构切（标题/API/函数），保留来源 URL 和版本。
- **Vector DB**：存 embedding + metadata（来源、版本、可信等级、是否废弃）。先过滤再排序。
- **Retriever**：不只是向量相似度，还要知道版本、环境、时间、地址。
- **Rerank**：重排序候选材料，减少"标题相似但内容不对"。
- **Citation**：答案必须能追溯回具体文档/段落，标注哪些是模型归纳、哪些证据不足。

关键金句：RAG 的核心不是让回答更长，而是让回答有来源、有版本、有边界。没有 citation 的 RAG 只是把幻觉从模型内部搬到了检索系统里。

### 3. 智能体（Agent）

> Handbook: https://aiweb3.school/zh/handbook/ai-basics/agent/

**第一性原理**：Agent 不是自主权本身，而是被约束的执行循环。目标、工具、状态、权限和停止条件缺一不可。

核心概念：
- **Tool Use**：工具从"会回答"变成"能做事"，但每类工具风险不同（读数据 vs 写数据库 vs 支付接口）。
- **Planning**：拆步骤是候选路线，不是授权。越靠近高风险动作，越需要系统规则介入。
- **State**：外置、可查询、可审计。不能只放 prompt 历史里。
- **Reflection**：自我检查提高质量，但不能替代外部验证。确定性检查才能承载风险。
- **Multi-Agent**：分工协作放大协调问题——上下文丢失、责任模糊、权限扩散。

关键金句：最危险的设计是让 Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限。

## Pipeline 串联

```
Knowledge Base → RAG(检索+引用) → Context(分层放置) → Agent(读状态+调工具+验证)
```

## Quiz 讲解（5 题串联关键工程直觉）

今天跳过答题，直接讲解。

### 1. Context 信任分层

场景：交易风险分析 Agent 把 dApp 官网声明、用户随口推荐、链上合约异常信号全部平铺在同一层，结果采信了不可信内容。

**缺陷**：没有对信息来源分层标注信任级别。Context Engineering 的核心是让模型知道每段信息的"身份"——系统指令 > 链上事实 > 用户意图 > 外部网页。正确做法：dApp 官网标"未验证外部内容"，用户推荐标"主观偏好"，链上未开源+owner 频繁变更标"高风险事实信号"。

### 2. RAG 搜到了但不对

场景：问 Uniswap V4 函数签名，搜到了 V2 的，看起来相似但错误。

**两个环节出错**：
- Chunking 没保留版本元数据，切分时丢掉了 V2/V3/V4 标记
- Retriever 只做纯向量搜索，没按版本过滤

修复：chunk 时带 `version` 字段，检索前 `filter(version="v4")`。

### 3. Agent 自我检查边界

场景：让 Agent 自我反思"有没有遗漏"，然后重新生成，直接展示结果。

**错误**：用模型的自我反思替代了确定性验证。模型可能换句话给同样的错误。正确做法：把投票建议拆成可验证字段（"是否所有反对理由都已引用来源？"），由代码而非模型检查。

### 4. Memory 隐性风险

场景：DeFi 助手记住用户上个月的 swap 池子地址，今天直接用历史记忆生成交易。

**错误**：Memory 不能替代实时授权。过去允许 ≠ 现在允许。池子地址要重新查、spender 要重新验证、approve 数量要当场确认。

### 5. 三章串联

| 层 | 关键问题 |
|---|---|
| Context | 提案正文、论坛讨论、链上投票——如何分层标注信任级别？ |
| RAG | 检索到的历史讨论如何附带时间戳和来源 URL，让 citation 可追溯？ |
| Agent | Planning 如何区分"安全只读分析"和"需用户授权的高风险步骤"？ |

## 收获

- 建立了 Context → RAG → Agent 三层 pipeline 的心智模型
- 掌握了 Context Engineering 的信任分层原则：每段信息必须标注来源身份和信任级别
- 理解了 RAG 的可靠性取决于证据链（版本 + 来源 + citation），不取决于向量库
- 明确了 Agent 安全设计的核心边界：自我反思 ≠ 安全校验，Memory ≠ 授权，Planning ≠ 执行许可
- 记住了一个安全红线：最危险的设计 = 模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限
