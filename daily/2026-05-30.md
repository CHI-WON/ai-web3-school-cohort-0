## B 阶段：Web3 基础三章

### Smart Contract（智能合约）
- 本质是"可验证执行"，不是代码很聪明
- 五核心：Storage（链上状态）、Event（可索引日志）、ABI（机器可读接口）、External Call（危险入口）、Upgrade（灵活但可被改）
- 完整调用链：前端 → 钱包签名 → RPC → 验证者 → EVM → receipt → 索引器
- 合约开发 ≠ 只写 Solidity——要理解钱包、RPC、浏览器、前端 SDK、event、索引层

### Security（安全）
- 链上安全 vs 传统：错误不可逆、代码公开、攻击者零成本试错
- 四个核心概念映射到 escrow：
  - Reentrancy：先改状态再转账（C-E-I）
  - Access Control：定义谁调 release/refund
  - Simulation：交易前预演结果
  - Monitoring：上线后异常告警

### Account Abstraction（账户抽象）
- 把账户从"一把私钥"变成可编程规则
- Session Key + Paymaster = Agent 权限底座
- AA 不是让 AI 更自由，而是把 AI 的自由用规则包起来
- 对应 proposal 里"单次 ≤5 USDC / 每天 ≤20 USDC"

## A 阶段：应用到 Proposal

### Escrow 合约伪代码
文件：`hackathon/escrow-contract.sol`

六个函数 + 完整状态机：
```
createTask → deposit → deliver → accept / refund / dispute
```

关键安全设计：
- **C-E-I 模式**：accept/refund 先改状态再转账（防重入）
- **Access Control**：deposit/accept/refund 只有 payer；deliver 只有 provider
- **双重超时**：deadline（交付截止）+ acceptWindow（验收窗口）
- **Dispute 状态**：资金锁在合约，由仲裁决定方向

### 流程图
文件：`hackathon/escrow-flowchart.md`

- 状态转换图
- 交互时序图（Alice → Agent → Escrow → Bob → Evaluator）
- 四个异常路径：超时、空交付、假翻译、恶意拒付
