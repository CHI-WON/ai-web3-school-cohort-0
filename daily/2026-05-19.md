# 2026-05-19 学习笔记

> 来源：Handbook → AI 基础 → [提示词（Prompt）](https://aiweb3.school/zh/handbook/ai-basics/prompt/)
> 模式：最小路径（~1h）
> 实验代码：`experiments/transaction_risk_prompt.py`

## 学习内容

### 1. Instruction — 任务规则的四段写法

- Prompt 是**软约束**，不是安全边界。真正边界由代码、权限、校验和审计承担。
- 四段写法：任务目标 / 可用输入 / 禁止行为 / 输出格式和失败格式
- **"解释"和"执行"要分开**：解释器可以分析风险，但不能替用户确认；代码助手可以生成 patch，但不能默认已通过测试。

### 2. Structured Output — 输出变成可机器校验的字段

- 核心逻辑：后续代码处理的是字段，不是散文。
- 标准字段设计：`action`（路由）/ `risk_level`（审批层级）/ `requires_human_approval`（guard 逻辑）/ `uncertainties`（审计追溯）
- 接上昨天的第三条原则：**输出变成可检查对象** → 能做自动化 eval、rule-based guard、快速定位问题来源。

### 3. 最小实践：交易风险摘要 Prompt

- 写了一个完整的 prompt 模板（四段结构 + JSON schema 输出）
- 3 组测试用例：
  1. 普通转账 → low risk（simulation 成功，意图匹配，无权限变更）
  2. 无限授权 → high risk（uint256 max，spender 不在白名单）
  3. 目标地址不匹配 → high risk（用户说换 UNI，实际换 SCAM 蜜罐代币）

## 收获

- 掌握了 Instruction 的四段拆法，能区分"解释"和"执行"的边界
- 理解了 Structured Output 不是形式要求，是工程上让 prompt 可测试、可校验的前提
- 建立了 Prompt + Structured Output + Guard 三层配合的心智模型
- 通过 3 个测试用例体会到：prompt 写好 + schema 约束 + guard 检查 = 安全地让 LLM 参与交易分析
